# 答题规范 + 数据一致性优先级 / Answer Norms & Data Consistency

> 所有 skill 生成回答、画图、判断事实时遵循的通用约定。可直接用，也可按需改写。

## 统一回答框架

所有技术问题按这个结构组织：

**类比开场 → 背景 → 原理（含图）→ 工程方案 → 数据结果 → 权衡 → 扩展**

- **类比开场**：用一句生活类比降低面试官的理解成本（从 `analogy-bank.md` 取）
- **背景**：这个问题在什么业务场景出现、为什么重要
- **原理**：机制 + 至少一张 Mermaid 图（从 `diagrams.md` 取）
- **工程方案**：你真实项目里怎么落地的（模块/类/设计）
- **数据结果**：量化成果（必须与 `profile.md` 锁定数据一致）
- **权衡**：为什么这么选、替代方案、风险、如果重来
- **扩展**：延伸追问准备

## 数据一致性优先级（强制）

当不同文件之间出现冲突时，按以下优先级判定「事实」：

1. `data/records/**/*.md` 的实际文件列表和内容（真实面试记录）
2. `data/pipeline.md`（面试管线）
3. `data/capability-profile.md`（能力画像）
4. README、skill 文档、本文件里的说明和示例

所有状态/分析/导航输出前必须做一次轻量一致性检查：
- 真实面试场次数 = `data/records/` 实际文件数，**不用**任何文档里的硬编码数字
- 管线统计（总投递/进行中/等待/终止）从 `pipeline.md` 当前表格重算
- 能力达标数从 `capability-profile.md` 当前表格重算
- 发现不一致 → 先输出「数据告警」，再基于实际文件给结果。**不静默修改**用户没请求修改的历史记录。

## Mermaid 语法规范

- sequenceDiagram：participant 用英文 ID + `as` 文字标签；必须加 `autonumber`
- flowchart：节点用 `["文字"]` 包裹；连线标签用 `-- "文字" -->` 格式
- 图要能直接渲染（是代码块，不是链接）

## 量化数据红线

- 回答中的所有数字必须能在 `profile.md` 的「量化数据锁定区」找到出处
- 被追问"这个数字怎么测的"时，答锁定区里登记的测量方法
- 临场编造数字 = 失分。不确定就说"这个我需要确认具体数字"，而不是猜
